What this does

The compiler implements the seven structural macros from SI §2.1 of the paper: 

exp(z)  ↦  eml(z, 1)
ln(z)   ↦  eml(1, exp(eml(1, z)))    = eml(1, eml(eml(1, z), 1))
x − y   ↦  eml(ln(x), exp(y))
−z      ↦  ln(1) − z
x + y   ↦  x − (−y)
1/z     ↦  exp(−ln(z))
x · y   ↦  exp(ln(x) + ln(y))

Plus derived macros for x², x/2, (x+y)/2, √x, x^y, log_b y, hypot(x, y), σ(x), sinh, cosh, tanh, arsinh, arcosh, artanh.

Trigonometric primitives are built from the complex EML grammar (EMLTermℂ). The closed constants i and π are shown as labelled leaves with their actual K-counts (their internal structure — the EMLRealizationℂ closure machinery — is hidden so the canvas stays legible). The result of a trig compile is read via .re or .im projection, shown above the metric tiles.

Trig witnesses are top-level constructions in the formal artefact — they don't compose with the real-fragment compiler in the browser tool. Path C′'s subst0-based widening (full-real-domain trig) is sealed in EML.Framework.Complex.Periodicity but isn't reproduced live here; the witnesses below are the local positive-side versions.

Tree sizes match the K_count_* theorems in the formalization (modulo the structural compiler's uniform-shape inflation vs. hand-tuned paper Table 4).

Source paper: A. Odrzywołek, All elementary functions from a single binary operator, arXiv:2603.21852. Lean formalization: github.com/nasqret/eml-formalization.

Builds the same EMLTerm trees as EML.Framework.Compilers.ELToEML.lean in the Lean artefact. No backend — everything runs in your browser.
